# 任务完成清单

## 每次任务完成后必须执行的检查

### 1. 类型检查 ✓
```bash
npm run type-check
```
或
```bash
npx vue-tsc --noEmit
```

**目的**: 确保 TypeScript 类型安全，无类型错误

**预期结果**: 无错误输出

### 2. 代码规范检查 ✓
```bash
npm run lint
```

**目的**: 确保代码符合 ESLint 规范

**预期结果**: 无错误，自动修复的问题已应用

### 3. 代码变更检查 ✓
```bash
git diff
```

**目的**: 检查所有变更内容，确保：
- 没有意外修改
- 没有调试代码残留
- 没有敏感信息泄露

### 4. 功能测试（可选）
如果修改了核心功能，建议：
```bash
npm run dev
```
在浏览器中手动测试相关功能

### 5. 提交变更
```bash
git add .
git commit -m "feat: 描述性的提交信息"
```

**提交信息规范**:
- `feat:` - 新功能
- `fix:` - 修复 bug
- `refactor:` - 重构
- `style:` - 样式调整
- `docs:` - 文档更新
- `test:` - 测试相关
- `chore:` - 构建或工具相关

## 特殊情况处理

### 如果类型检查失败
1. 检查错误提示
2. 修复类型定义或类型断言
3. 不要使用 `@ts-ignore` 除非绝对必要
4. 重新运行类型检查

### 如果 ESLint 检查失败
1. 查看错误详情
2. 运行 `npm run lint` 自动修复
3. 手动修复无法自动修复的问题
4. 不要禁用规则除非有充分理由

### 如果构建失败
1. 检查构建错误日志
2. 确保所有依赖已安装 (`npm install`)
3. 清除缓存重试：`rm -rf node_modules dist && npm install`

## 最佳实践

1. **增量提交**: 每完成一个小功能就提交一次
2. **描述性消息**: 提交信息要清晰描述做了什么
3. **测试先行**: 修改核心逻辑前先思考如何测试
4. **代码审查**: 提交前检查 `git diff` 输出
5. **保持整洁**: 不提交临时文件、调试代码
